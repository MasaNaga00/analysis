# state_logic.py ドキュメント — 累積使用率 固定しきい値運用の状態判定

部品出庫データ（datamart_A）と台帳（Excel）を入力に、累積使用率の固定しきい値運用における
各部品の状態を判定し、「要確認インボックス」と「Tableau用監視テーブル」を出力するモジュール。

設計の前提（台帳をマスタにした判断の記録、部品単位へのアラート集約、機種ごと対象外など）は
台帳テンプレートの「説明」シートおよび設計検討の経緯に従う。本書はその実装仕様にあたる。

---

## 1. 何をするものか（全体像）

毎月の運用サイクルの中で、このモジュールが担うのは「機械判定」の部分だけ。

```
datamart_A ──┐
             ├─→ state_logic.py ──→ 要確認インボックス.csv ──→ 人がレビューし台帳に追記
台帳.xlsx ───┘         │
                       └────────→ tableau_監視テーブル.csv ──→ Tableauで可視化
```

- 判断（原因の調査、処置区分の選択、再アラート閾値Yの決定）は人が台帳に書く。
- 本モジュールは台帳を読んで「各部品の現在の有効な条件」を機械的に解決し、
  当月の累積率と突き合わせて鳴らすかどうかを決めるだけ。判断は一切しない。
- 状態（正常/監視中/要確認/終了）はどこにも保存しない。毎回、台帳＋パネルから計算で導く。
  → 台帳が唯一のマスタであり、台帳を直せば過去の状態も再計算で正しくなる。

## 2. 入力

### 2.1 パネル（datamart_A のエクスポート）

縦長の月次パネル。1行 = 事業コード × 開発コード × 部番 × 販社 × 年月。
CSV / Excel どちらも可（`CONFIG["panel_path"]` の拡張子で自動判別）。

必要な列（実データの列名は `CONFIG["cols"]` でマッピングする）:

| 内部名 | 既定の列名 | 役割 |
|---|---|---|
| biz | 事業コード | キー |
| dev | 開発コード | キー（機種） |
| part | 部番 | キー |
| dist | 販社 | キー |
| ym | 年月 | YYYYMM（整数/文字列/日付様式いずれも可。内部でYYYYMM整数に正規化） |
| monthly_use | 月次使用数 | 監視する生カウント |
| cum_sales | 累積販売台数 | 率の分母 |
| cum_use | （任意） | 累積使用数の列が元データにあれば列名を指定。無ければ `None` で月次使用数から累積を計算 |

### 2.2 台帳（Excel）

台帳テンプレートの「台帳」シート。追記専用ログで、1行 = 1回の確認イベント。
読み込み時に判定年月・再評価年月を YYYYMM 整数へ、条件パラメータY を数値へ正規化する。

本モジュールが参照する列: 事業コード / 開発コード / 部番 / 判定年月 / 再アラート条件タイプ /
条件パラメータY / 再評価年月 / 記録日（同月内の順序づけ）。
処置区分・確認者・メモは人間向けの記録であり、判定には使わない
（判定に効くのは「再アラート条件タイプ」と「Y」。処置区分は条件タイプを選ぶためのメニュー）。

### 2.3 CONFIG（つまみ一覧）

| キー | 既定値 | 意味 |
|---|---|---|
| ledger_path / ledger_sheet | テンプレートxlsx / "台帳" | 台帳の場所 |
| panel_path / panel_sheet | None / 0 | パネルの場所。None だとデモ実行 |
| cols | （上表） | 列名マッピング。cum_use だけ None 可 |
| base_threshold_pct | 2.0 | 既定の基準閾値 X（%） |
| threshold_overrides | {} | X の上書き。キーはタプルで `("事業",)` `("事業","開発")` `("事業","開発","部番")` の3階層。**具体的なキーが優先**（部番指定 > 機種指定 > 事業指定） |
| margin_pct | 0.5 | インボックスに出す「提案Y下限」= 観測率 + margin。**実際のYは人が台帳に書く**（これは提案値の計算にだけ使う） |
| min_denominator | 0 | 販社系列がトリガ資格を持つ累積販売台数の下限。0=全販社に資格。合算系列は常に資格あり |
| fill_zero_months | True | 月次0補完のオン/オフ |
| fill_to_global_max | False | 補完範囲を各系列の最終月まで(False)か、パネル全体の最新月まで(True)か |
| asof_ym | None | 判定の基準月。None ならパネル最新月。過去月を指定すれば「その月時点の判定」を再現できる |
| out_inbox / out_tableau | CSV名 | 出力先（UTF-8 BOM） |
| machine_all_part_token | "機種全体" | 機種終了行の部番に入る特別値 |

## 3. 処理の流れと、各段階でやっている評価

### 3.1 前処理: 月次0補完（`_fill_zero_months`）

各系列（事業×開発×部番×販社）について、初出月〜最終月の間で抜けている年月を0行で埋める。

- 月次使用数 = 0（「その月は出庫が無かった」という証拠として扱う）
- 累積販売台数 = 前方補完（直前月の値で据え置き）
- 累積使用数（元データに列がある場合）= 前方補完

DBから1系列1月1行で取れている限り発火しない安全網。発火した場合の副作用は
「補完月の分母が直前月で据え置かれるため、その月の率がごくわずかに高めに出る」のみ。

`fill_to_global_max=True` にするとパネル全体の最新月まで埋めるが、既定は各系列の
観測範囲の内側だけ（未発売・対象外の期間に0行を作らないため）。

### 3.2 系列レートの計算（`series_rates`）

判定の材料になる「系列×月の累積率テーブル」を作る。系列は2種類:

1. **販社別系列**: パネルの行そのまま。`系列 = 販社名`
2. **合算系列**: 同一部品の全販社を月ごとに合計（累積使用数・累積販売台数とも合計してから率を取る）。`系列 = "合算"`

各系列に対して:

- `累積率(%) = 100 × 累積使用数 ÷ 累積販売台数`（分母0なら NaN）
- `トリガ資格(eligible)`: その系列がアラートの引き金になれるか。
  - 合算系列: 分母 > 0 なら常に資格あり
  - 販社系列: 分母 ≥ `min_denominator` のときだけ資格あり
  - **資格の無い系列も率は計算され、Tableauには出る**（表示はするがトリガにしない、という設計。
    低頻度部品で小さな販社の率が跳ねて誤トリガするのを防ぐためのつまみ。既定はオフ）

### 3.3 機種終了の解決（`machine_end_set`）

台帳から「再アラート条件タイプ=機種終了」または「部番=機種全体」の行を集め、
(事業, 開発) → 発効年月 の辞書を作る。同一機種に複数行あれば**最も早い判定年月**が発効月。

発効月以降、その機種に属する**全部品・全系列**が判定対象外になる。
発効月より前の月は通常どおり判定される（過去に遡って消さない。Tableauで履歴は見える）。

### 3.4 部品の状態解決（`resolve`） — 判定の心臓部

任意の部品 (事業, 開発, 部番) と月 t に対し、「月 t 時点の状態・当月閾値・次の判定種別」を返す。
評価は次の優先順位で行う:

```
1. 機種終了が発効済み(発効月 ≤ t)か？        → 状態=終了機種。判定しない
2. 台帳に t 以前の行が無いか？               → 状態=未記録。閾値=基準X。次に鳴れば「初回」
3. t 以前の最新行の条件タイプは何か？
   ├ 監視終了   → 状態=終了単位。判定しない
   ├ 保留       → 状態=要確認保留。翌月もインボックスに残り続ける
   │             （閾値は、それ以前の「率超過」行のYがあればそれ、無ければ基準X）
   └ 率超過     → 状態=監視中。閾値=その行のY。次に鳴れば「再」
4. 最新行の再評価年月が t 以下なら「再評価到来」フラグを立てる
```

ポイント:

- **「t 以前の最新行」で評価する**ため、`asof_ym` に過去月を入れれば当時の判定を完全に再現できる
  （台帳は追記専用なので、後から書いた行が過去の判定を変えることはない。
  同じ判定年月に複数行ある場合は記録日の新しい方を採る）。
- **基準閾値 X の解決**は `threshold_overrides` を具体側から探す:
  (事業,開発,部番) → (事業,開発) → (事業,) → 既定値、の順で最初に見つかったもの。
- 保留の挙動: 条件を満たすかどうかに関係なく毎月インボックスに載り続ける。
  「調査中なので黙らせる」ではなく「調査中なので見え続けさせる」設計。放置の検知漏れを防ぐ。

### 3.5 要確認インボックスの生成（`build_inbox`）

基準月 T（既定はパネル最新月）について、全部品を `resolve` で評価し、
**人がレビューすべき部品だけ**を一覧にする。部品が載る条件は次の3つのいずれか:

| 理由 | 条件 |
|---|---|
| 初回 / 再 | トリガ資格があり率が NaN でない系列のうち、**どれか1つでも**当月閾値以上 |
| 保留継続 | 最新行が「保留」（率に関係なく毎月載る） |
| 再評価 | 最新行の再評価年月 ≤ T（率が動かなくても時間で載せる） |

載らない条件:

- 終了単位・終了機種
- **当月 T にすでに記録済み**（最新行の判定年月 = T、かつ保留でない）
  → 月の途中で台帳に書いた直後に再実行しても、同じ部品が二重に出ない（冪等性）。
    保留だけは例外で、当月記録でも載り続ける。

各行に付く情報と、その決め方:

- **観測率 / 判定元**: 閾値を超えた系列のうち**最大の累積率とその系列名**。
  どの系列も超えていない場合（保留継続・再評価で載った場合）は、資格ありの系列の最大率を参考値として載せる。
- **判定種別**: 台帳に行が無ければ「初回」、あれば「再」。
- **提案Y下限**: 観測率 + `margin_pct`。**人がYを決める際の下限の目安**であり、
  これ未満のYを書くと翌月即再発火する、という意味のガードレール。自動でYを決める機能ではない。

このCSVの列は台帳の「自動」列と同じ並びなので、レビュー後はこの行を台帳にコピーし、
処置区分以降の人間記入列を埋めれば1件の記録が完了する。

### 3.6 Tableau用監視テーブルの生成（`build_tableau`）

系列×月の全行に、判定結果を注釈として付けた縦長テーブル。各行(部品×月)について
`resolve` を呼び、その月時点の閾値と状態を計算する。列と意味:

| 列 | 意味 |
|---|---|
| 系列 | `合算` または販社名 |
| 累積率 | その系列・その月の累積使用率(%) |
| 累積使用数 / 累積販売台数 | 率の分子・分母（Tableauのツールヒント用） |
| トリガ資格 | この系列が引き金になれるか（3.2参照） |
| 当月閾値 | その月時点で部品に有効な閾値。台帳の記入に従い X→Y→Y' と階段状に変わる。**部品で1本**（系列によらず同じ値） |
| 閾値超 | 資格あり ∧ 率が当月閾値以上（系列ごとに計算） |
| 状態 | 終了 / 要確認 / 監視中 / 正常（下記） |
| 判定点 | その月に台帳の記録があるか（=人が確認した月のマーカー） |
| 台帳記録あり | **部品レベル**: その部品に台帳の行があるか（機種終了の機種に属する場合も True）。「判定済みだけ表示」スイッチの実体 |
| 最新状態 | **部品レベル**: 最新月時点の状態。ヒートマップの行フィルタはこちらを使う（時系列の`状態`で絞ると月単位で帯が欠ける） |
| 注目度 | **部品レベル**: 最新月における 累積率÷当月閾値 の最大（資格あり系列のみ）。1.0超=どこかが閾値超。未判定部品を含めた行ソート用。終了部品は NaN |

状態の決め方（部品×月単位。同じ部品・同じ月なら全系列で同じ状態）:

```
終了      機種終了 or 監視終了が発効済み
要確認    台帳にその月の記録がある(判定点) または どれかの系列が閾値超
監視中    台帳に過去の記録があり、当月はどの系列も閾値未満
正常      台帳に記録が無く、当月どの系列も閾値未満
```

つまり「要確認」は**鳴っている月**、「監視中」は**人が条件を設定して静観している月**を表す。
インボックスと違い、当月記録済みでも「要確認」のまま表示する
（その月に発火・対応した事実を可視化として残すため。冪等性の除外はインボックス側だけ）。

## 4. 使い方

### 4.1 デモ実行（実データなし）

```bash
python state_logic.py
```

`panel_path=None` のままだと内蔵の疑似データ＋台帳記入例で動き、
当月の系列別レート・インボックス・閾値の階段をコンソールに表示する。
仕組みの確認、Tableauワークブックの試作（デモCSVを使う）に使える。

### 4.2 実データでの実行

1. `CONFIG["panel_path"]` に datamart_A のエクスポートのパスを設定
2. `CONFIG["cols"]` を実データの列名に合わせる（`cum_use` は列が無ければ `None` のまま）
3. `CONFIG["base_threshold_pct"]`（X）を現場の値に。事業ごとに変えるなら `threshold_overrides`
4. `python state_logic.py` → `要確認インボックス.csv` と `tableau_監視テーブル.csv` が出る

### 4.3 毎月の運用手順

```
1. datamart_A の最新エクスポートを所定パスに置く
2. python state_logic.py
3. 要確認インボックス.csv を開き、各部品を調査
4. 台帳.xlsx に1行ずつ追記（自動列はインボックスからコピー、処置区分以降を記入）
5. （任意）再実行して、記録済みの部品がインボックスから消えることを確認
6. tableau_監視テーブル.csv をTableauで更新
```

### 4.4 過去月の判定を再現する

`CONFIG["asof_ym"] = 202603` のように基準月を指定すると、その月時点のインボックスを再現できる。
台帳が追記専用である限り、いつ実行しても同じ結果になる（監査・振り返り用）。

## 5. 仕様上の注意・既知の限界

- **累積率は下がらない**: 累積比のため、対策が効いても率は横ばいになるだけ。
  このツールで「改善」は確認できない（役割は要注目ゲート。改善確認は並行検討のCUSUM側）。
- **Yは部品で1本**: 販社別に別の感度は持てない（アラート集約の代償）。
  基準Xだけは `threshold_overrides` で部番単位まで differentiable。
- **経過月の交絡**: 古い機種ほど累積率が自然に高く、鳴りやすい。
  本ツール側の対処は「末期・対象外」「機種ごと対象外」を台帳で選ぶ運用のみ。
- **同月複数行**: 同じ部品・同じ判定年月に複数行書かれた場合は記録日の新しい行が有効。
  記録日が空だと順序が不定になるため、記録日は必ず入れること。
- **台帳のYの妥当性チェックはしない**: Y < 観測率 のような「翌月即再発火する記入」も
  そのまま受け入れる（提案Y下限はあくまで目安）。記入ルールの統制は運用側で行う。
- **部番=機種全体 の行は部品の履歴に混ぜない**: 機種終了の解決にだけ使い、
  個別部品の閾値履歴（3.4のステップ3）からは除外している。
